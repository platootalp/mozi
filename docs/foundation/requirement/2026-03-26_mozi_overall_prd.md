## 2. 概述

### 2.1 背景与目的

**背景**

当前AI编程助手市场呈现快速发展态势，竞品如Cursor、OpenCode、OpenClaw等各有特色：
- Cursor采用云端IDE模式，Plan Mode提供显式规划但需用户手动触发
- OpenCode采用本地CLI模式，双主Agent分离设计
- OpenClaw采用本地CLI模式，ReAct隐式规划，无自适应路由

**目的**

Mozi旨在打造一款**本地优先**的AI Coding Agent，通过**自适应复杂度路由**和**Orchestrator中心化调度**实现智能化任务处理，解决现有方案在规划模式、多Agent协作、记忆管理等方面的痛点。

**核心差异化**：
1. 自适应规划：无需用户显式选择模式，系统自动根据复杂度路由
2. Orchestrator中心化调度：区别于OpenCode的层级委托和Cursor的并行执行
3. 三层记忆+自动分层+RAG检索：竞品多为单一存储或需手动配置
4. 完整四层安全防护：含HITL审批机制

### 2.2 目标用户

| 用户群体 | 使用场景 | 核心需求 |
|----------|----------|----------|
| 个人开发者 | 日常编码、代码重构、bug修复 | 快速响应、低延迟、本地数据安全 |
| 小型开发团队 | 协作开发、代码审查、技术债务清理 | 多Agent协作、任务分解、审计追溯 |
| 企业用户 | 大型项目维护、代码库分析、自动化工作流 | 安全合规、HITL审批、水平扩展 |
| DevOps工程师 | CI/CD集成、自动化脚本、基础设施代码 | CLI集成、Webhook、API接口 |

### 2.3 解决的问题

| 问题 | 现状（竞品） | Mozi解决方案 |
|------|--------------|--------------|
| 规划模式需手动选择 | Cursor/OpenCode需用户显式切换 | 自适应复杂度评估，自动路由 |
| 多Agent协作复杂 | OpenCode层级委托，职责不清 | Orchestrator中心化DAG调度 |
| 记忆管理单一 | OpenClaw本地Markdown，无分层 | 三层记忆+自动分层+RAG检索 |
| 安全模型不完整 | 缺少HITL机制 | 四层安全防护+强制审批 |
| 上下文长度限制 | 无智能压缩 | 上下文Token 80%触发压缩+摘要生成 |

### 3.3 核心用例

#### UC-1: 简单代码修改
| 用例ID | UC-1 |
|--------|------|
| 用例名称 | 简单代码修改 |
| 参与者 | 开发者 |
| 触发条件 | 用户请求修改单个文件或修复简单bug |
| 前置条件 | 用户已启动Mozi CLI并进入项目目录 |
| 基本流程 | 1. 用户输入修改需求<br>2. 系统识别为SIMPLE复杂度<br>3. 系统走FastPath直接执行<br>4. 系统返回修改结果 |
| 备选流程 | 2a. 需求不明确→系统询问澄清<br>4a. 编辑冲突→自动重试或HITL |
| 后置条件 | 文件已修改，变更已记录 |
| 复杂度 | SIMPLE (≤40分) |
| 业务规则 | 1. 简单任务直接执行，无需生成计划<br>2. 单次编辑不超过5个文件<br>3. 不涉及跨模块依赖修改 |

#### UC-2: 复杂任务分解执行
| 用例ID | UC-2 |
|--------|------|
| 用例名称 | 复杂任务分解执行 |
| 参与者 | 开发者、审批者（可选） |
| 触发条件 | 用户请求跨模块重构或大型功能开发 |
| 前置条件 | 用户已启动Mozi CLI |
| 基本流程 | 1. 用户输入任务需求<br>2. 系统识别为COMPLEX复杂度 (>70分)<br>3. Orchestrator生成DAG计划<br>4. 用户确认/修改DAG计划（可选）<br>5. 多Agent并行/串行执行<br>6. 结果聚合返回 |
| 备选流程 | 4a. 涉及敏感操作→触发HITL审批<br>5a. Agent执行失败→DAG重新调度<br>5b. 连续失败→终止并报告 |
| 后置条件 | 所有子任务完成，结果已汇总 |
| 复杂度 | COMPLEX (>70分) |
| 业务规则 | 1. DAG节点数不超过20个<br>2. 单节点超时时间可配置<br>3. 并行节点数不超过5个 |

#### UC-3: 外部工具集成
| 用例ID | UC-3 |
|--------|------|
| 用例名称 | 外部工具集成 |
| 参与者 | 开发者 |
| 触发条件 | 用户请求使用MCP工具（如git、数据库等） |
| 前置条件 | MCP服务已配置 |
| 基本流程 | 1. 用户请求使用外部工具<br>2. 系统通过MCP Client调用工具<br>3. 工具执行结果返回<br>4. 结果聚合到Agent上下文 |
| 备选流程 | 2a. MCP服务不可用→降级或禁用该服务<br>3a. 工具权限不足→HITL审批 |
| 后置条件 | 工具调用结果已记录 |
| 复杂度 | 任意 |
| 业务规则 | 1. MCP工具调用必须通过MCP Client代理<br>2. 禁止直接shell调用MCP服务 |

#### UC-4: 会话管理与恢复
| 用例ID | UC-4 |
|--------|------|
| 用例名称 | 会话管理与恢复 |
| 参与者 | 开发者 |
| 触发条件 | 用户需要恢复历史会话或创建子会话 |
| 前置条件 | 存在历史会话记录 |
| 基本流程 | 1. 用户请求恢复会话<br>2. 系统从冷存储加载历史上下文<br>3. 上下文解压重建<br>4. 会话恢复继续 |
| 备选流程 | 2a. 会话已归档→从归档恢复<br>3a. 上下文超限→触发压缩 |
| 后置条件 | 会话状态完整恢复 |
| 复杂度 | 任意 |
| 业务规则 | 1. 会话恢复后需重新评估当前任务复杂度<br>2. 子会话共享父会话历史但独立执行 |

#### UC-5: 敏感操作审批
| 用例ID | UC-5 |
|--------|------|
| 用例名称 | 敏感操作审批 |
| 参与者 | 开发者、审批者 |
| 触发条件 | Agent执行敏感操作（bash、文件覆盖等） |
| 前置条件 | 操作被识别为敏感 |
| 基本流程 | 1. Agent触发敏感操作<br>2. 系统暂停执行，发送审批请求<br>3. 审批者 approve/reject<br>4. 系统根据决策继续/终止 |
| 备选流程 | 3a. 审批超时→默认拒绝<br>3b. 紧急模式→跳过审批记录原因 |
| 后置条件 | 审批结果已记录到审计日志 |
| 复杂度 | 任意（横切关注点） |
| 业务规则 | 1. 敏感操作必须包含操作原因说明<br>2. 审批结果不可篡改<br>3. 紧急模式需记录跳过的原因 |

### 3.4 边界与约束场景

| 场景 | 描述 | 系统行为 |
|------|------|----------|
| BC-1 | 用户请求删除整个项目目录 | 触发HITL审批，操作前必须用户二次确认 |
| BC-2 | 用户请求提交代码到外部仓库 | 仅在明确授权的仓库白名单内允许 |
| BC-3 | 用户请求执行不可逆的数据库操作 | 必须显示预估影响范围并等待确认 |
| BC-4 | 并发修改同一文件 | 使用哈希锚定检测冲突，冲突时触发重试或HITL |
| BC-5 | 超出Token限制的长对话 | 自动触发上下文压缩，保留关键决策点 |
