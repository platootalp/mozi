## 2. 范围问题详细说明

### 2.1 超出 PRD 范畴的内容

#### Critical Issue #1: API 契约（Section 6）不应出现在 PRD 中

**问题描述**: Section 6 "API 契约" 共约 300 行，包含：
- 完整的 REST API 定义（6.2.1 ~ 6.2.6）
- WebSocket 流式 API 规范（6.3）
- 请求/响应 JSON 示例

**问题分析**: API 契约是**技术实现规格**，属于**架构设计或开发文档**范畴，而非产品需求。PRD 应聚焦于"需要什么功能"，而非"如何暴露接口"。

**建议处理**:
- 将 Section 6 拆分到独立的 `API_Contract.md` 或纳入架构文档
- PRD 中仅保留对 API 需求的简要描述（如"需要提供 RESTful API 支持第三方集成"）

#### Critical Issue #2: 排期建议（Section 10）属于项目管理而非产品需求

**问题描述**: Section 10 包含：
- 详细的迭代计划（v1.0 MVP / v1.1 / v1.2）
- 精确的工期估算（10周 x 3）
- 资源分配表（6人团队配置）
- 里程碑日期（精确到 2026-06-04）

**问题分析**: 排期是**项目管理职责**，不是产品需求。PRD 的作用是定义"做什么"，而非"谁做"和"多久做完"。

**建议处理**:
- 将 Section 10 拆分到 `Iteration_Plan.md` 或 `Sprint_Planning.md`
- PRD 中仅保留优先级分类（P0/P1/P2 功能清单）

#### Critical Issue #3: 技术选型重复

**问题分析**: PRD Section 4 的各层需求（Ingress/Orchestrator/Capabilities/Infrastructure）与架构文档 Section 2-9 存在大量技术细节重复。

**示例**:

| PRD 中的位置 | 重复内容 | 应归属 |
|--------------|----------|--------|
| Section 4.2 接入层需求 | CLI工具、Web UI、API Gateway等技术描述 | 架构文档 |
| Section 4.3 编排层需求 | 意图识别、复杂度评估等技术规格 | 架构文档 |
| Section 4.4 能力层需求 | MCP集成、Skills引擎等技术细节 | 架构文档 |
| Section 4.5 基础设施层需求 | 存储层设计、技术选型等 | 架构文档 |

### 2.2 与主架构文档的职责划分问题

两个文档的关系应如下：

```
PRD (需求文档)
├── "做什么" - 产品需要什么功能
├── "为什么" - 用户痛点和价值
├── "做到什么程度" - 验收标准
└── "优先做什么" - 优先级排序

架构文档 (设计文档)
├── "怎么做" - 技术方案
├── "用什么做" - 技术选型
├── "组件关系" - 架构图和模块关系
└── "边界在哪" - 接口定义
```

**当前问题**: PRD 大量侵入"怎么做"的领域，而"做什么"的描述不够深入。
